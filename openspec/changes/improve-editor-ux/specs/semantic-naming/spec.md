# 语义化节点命名规范

## 新增需求

### 需求:语义化递增命名

系统必须为新创建的节点生成语义化递增名称，格式为 `<前缀>_<数字>`。

#### 场景:function 节点命名
- **当** 用户创建 function 类型节点
- **那么** 节点名称为 `fun_1`、`fun_2`、`fun_3`...

#### 场景:tool 节点命名
- **当** 用户创建 tool 类型节点
- **那么** 节点名称为 `tool_1`、`tool_2`、`tool_3`...

#### 场景:subgraph 节点命名
- **当** 用户创建 subgraph 类型节点
- **那么** 节点名称为 `subgraph_1`、`subgraph_2`、`subgraph_3`...

#### 场景:不同类型独立计数
- **当** 用户依次创建 fun_1、tool_1、fun_2
- **那么** 各类型计数独立，互不干扰

---

### 需求:计数器不回收删除编号

系统必须在节点被删除后不回收其编号，避免命名冲突。

#### 场景:删除后计数器不减少
- **当** 用户创建 fun_1、fun_2、fun_3，然后删除 fun_2
- **那么** 下一个 function 节点命名为 fun_4

#### 场景:删除后创建不重用编号
- **当** 用户删除 fun_5 后创建新 function 节点
- **那么** 新节点命名为 fun_6（不是 fun_5）

---

### 需求:加载项目时重建计数器

系统必须在加载项目时从现有节点名称解析重建计数器。

#### 场景:加载语义命名项目
- **当** 用户加载包含 fun_1、fun_5、tool_3 的项目
- **那么** 计数器重建为 { function: 5, tool: 3, subgraph: 0 }
- **且** 下一个 function 节点命名为 fun_6

#### 场景:加载 UUID 命名项目
- **当** 用户加载使用旧 UUID 命名的项目
- **那么** 计数器保持为 0
- **且** 新创建节点从 fun_1 开始命名

#### 场景:加载混合命名项目
- **当** 用户加载同时包含语义名称和 UUID 名称的项目
- **那么** 仅语义名称参与计数器计算

---

### 需求:节点 ID 仍使用 UUID

系统必须保持节点 ID 为 UUID 格式，仅修改节点名称。

#### 场景:节点 ID 格式
- **当** 用户创建新节点
- **那么** 节点 `id` 为 UUID（如 `a1b2c3d4-e5f6-...`）
- **且** 节点 `data.name` 为语义名称（如 `fun_1`）

---

### 需求:代码生成使用节点名称

系统必须在导出的 Python 代码中使用节点名称而非 ID。

#### 场景:边连接使用节点名称
- **当** 用户连接 fun_1 和 tool_1
- **那么** 生成的代码为 `graph.add_edge("fun_1", "tool_1")`

#### 场景:入口点使用节点名称
- **当** 用户设置 fun_1 为入口点
- **那么** 生成的代码为 `graph.set_entry_point("fun_1")`

#### 场景:生成的代码无 UUID
- **当** 用户导出 Python 代码
- **那么** 代码中无 UUID 字符串

---

### 需求:名称格式验证

系统必须确保生成的节点名称是有效的 Python 标识符。

#### 场景:名称符合 Python 标识符规则
- **当** 系统生成节点名称
- **那么** 名称匹配正则表达式 `/^[a-zA-Z_][a-zA-Z0-9_]*$/`

#### 场景:生成的 Python 代码可执行
- **当** 用户导出 Python 代码
- **那么** 代码可通过 `python -m py_compile` 验证无语法错误
